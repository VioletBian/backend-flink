## 1. 目标与背景

本次实现的核心目标不是“立刻把 Python pandas 引擎 100% 翻译成纯 Java/Flink”，而是先把下面三件事落稳：

1. 新建一个与 `backend` 平行的 `backend-flink` 独立后端目录。
2. 让 Java 服务对外暴露与 Python 后端兼容的 REST 路由和 Mongo 存储契约。
3. 在“同一输入表 + 同一 pipeline JSON”下，先通过 canonical DSL + Python reference runner 兜底，保证结果语义不漂移，再逐步把可控算子迁移为 Flink 原生执行。

也就是说，当前阶段的优先级是：

- 先统一 DSL
- 先统一接口
- 先统一结果
- 再逐步替换执行内核

## 3. backend-flink 当前架构

当前执行链路是：

1. Java 接收请求
2. `PipelineContractNormalizer` 把 legacy JSON 归一为 canonical DSL
3. `PipelineRunService` 按 `enableParallel` 补齐每个 step 的 runtime `execution`
4. `StagePlanner` 把逻辑 step 展开成 spec，并按当前中间表形状动态切 segment
5. 每个 segment 决定走 `FLINK_NATIVE` 还是 `PYTHON_FALLBACK`
6. Java native 段当前实际执行 `rename / formatter / date_formatter`
7. 其余段按 stage 粒度调用 Python reference runner
8. `debug=true` 时，复用 prepared pipeline 补回 `step_snapshots`

当前关键类职责如下：

- `PipelineController`
  - 对外暴露 REST API
- `PipelineContractNormalizer`
  - 统一字段方言，拒绝超出首期合同的算子
- `OperatorCapabilityRegistry`
  - 描述每个算子的行切分/列切分/屏障特征
- `PipelineRunService`
  - 串联 execution enrich、segment 规划、native/fallback 执行和 debug 回填
- `StagePlanner`
  - 生成 `prepared_pipeline`、`OperationSpec`、`StagePlan`、`PipelinePlan`
- `HybridPipelineService`
  - 作为 `/run` 门面服务转发到 `PipelineRunService`
- `NativeStageExecutor`
  - 承接当前最保守的 Java/Flink 原生 stage
- `PythonFallbackBridge`
  - 以 stage 粒度调用 `backend/scripts/reference_runner_cli.py`
- `PipelineStoreService`
  - 存取 Mongo 版本数据
- `BatchTableEnvironmentFactory`
  - 提供 Flink batch Table / DataStream 环境工厂

## 4. 当前“已支持”的真实含义

这里必须明确区分两种“支持”：

### 4.1 接口级支持

指的是：

- Java 服务可以接受该算子的 DSL
- 可以进行 canonicalization / plan / save / load
- 可以通过 Python bridge 正确执行整条 pipeline

在这个意义下，当前首期合同内算子都“可用”：

- `filter`
- `rename`
- `aggregate`
- `sort`
- `tag`
- `col_assign`
- `col_apply`
- `series_transform`
- `formatter`
- `date_formatter`

### 4.2 Flink 原生执行级支持

指的是：

- 该算子已经真正由 Java/Flink 执行
- 不是只靠 Python bridge 兜底
- 已独立验证与 Python 结果完全一致

**这一层当前已经开始落地，但仍然只覆盖很小的保守子集。**

更准确地说：

- 现在 backend-flink 已经能在 Java 侧真实执行一部分 stage
- 当前真实 native 子集是：
  - `rename`
  - `formatter`
  - `date_formatter`
- 其他算子或不安全 stage 仍然由 Python fallback 保证结果一致性
- 所以当前的一致性来源是：
  - 小部分来自 Java/Flink native 实现
  - 更大部分仍来自 Python fallback 兜底

## 5. 当前算子适配矩阵

### 5.1 当前已经原生执行的算子

以下算子现在已经会在满足 stage 条件时走 Java/Flink native path：

- `rename`
- `formatter`
- `date_formatter`

但要注意：

- `rename` 只支持 rows/serial native，不参与 columns native
- `formatter` / `date_formatter` 支持 rows/columns native
- 这个 native 范围是“当前已实现”，不是最终目标范围

### 5.2 当前 planner 仍会规划、但执行时回退 Python 的算子/场景

以下情况当前仍然由 Java planner 切段，但执行时统一交给 Python fallback：

- `filter`
- `tag`
- `col_assign`
- `aggregate`
- `sort`
- `col_apply`
- `series_transform`

### 5.3 当前明确适配性不足的算子/场景

以下情况从实现角度上仍只能靠 Python 保证一致性：

- `col_assign`
  - `method=lambda`
  - `method=vectorized` 里带复杂表达式的场景
- `col_apply`
  - 当前全部场景
- `series_transform`
  - 当前全部场景
- `filter`
  - 带复杂 pandas query 语义的条件表达式
- `tag`
  - 带复杂 `eval` 条件表达式的场景

### 5.4 当前适配性不够好的部分

下面这些是本次你特别强调需要在文档里写清楚的“目前走 java-flink 还不支持 python 一致性效果 / 适配性不够好”的重点：

#### A. `col_apply`

问题：

- 当前 Python 版本允许 `method` 直接是任意 callable 表达式
- 这类逻辑可能接收 `DataFrame` 或 `Series`
- Java/Flink 侧很难在不收紧 DSL 的前提下做通用纯原生翻译

当前状态：

- planner 会把它切成独立 fallback stage
- backend-flink 当前没有 Flink 原生实现

风险：

- 如果未来强行用 Java 表达式子集替代，会和现有 Python callable 语义发生漂移

#### B. `col_assign` 的 `lambda` 模式

问题：

- 当前 Python 版本允许 `lambda x: ...` 直接访问 row 级对象
- 实际表达能力是 Python 本身，而不是一个受限的中间表达式语言

当前状态：

- 当前无论 `vectorized` 还是 `lambda`，都还没有 Java 原生实现
- 运行时统一回退 Python

风险：

- Java/Flink 侧若只支持简单算术表达式，则无法保证与 Python `lambda` 完全同义

#### C. `series_transform`

问题：

- Python 版本的 `transform_expr` 可以是任意对 `pd.Series` 的 callable
- 不仅可以做 `shift/diff/rolling`
- 还可以做更复杂的 pandas / numpy 逻辑

当前状态：

- 当前 `series_transform` 整体仍回退 Python
- Java planner 只把它视作 columns-capable 的规划对象，不代表已经具备原生执行

风险：

- 后续如果要做 native 化，仍然需要非常明确的表达式分型与白名单能力检测

#### D. `aggregate`

问题：

- 虽然它在优化目标上适合做两阶段聚合
- 但当前 Java 服务里还没有真正把 aggregate 下推到 Flink 执行

当前状态：

- 当前只会把它当作 barrier stage
- 实际结果仍由 Python reference runner 生成

风险：

- native aggregate 一旦真正落地，仍需要重点校验：
  - rename 语义
  - 输出列顺序
  - groupby/reset_index 行为
  - 空值与 dtype 行为

#### E. `sort`

问题：

- Python 侧已改为稳定排序 `mergesort`
- 这样可以让相同 sort key 下的顺序可复现

当前状态：

- planner 已将其视为 barrier
- 但 Java/Flink 侧还没有真正实现稳定排序后的最终结果重建

风险：

- 原生执行时若没有显式 `_row_id` + stable ordering 策略，很容易出现与 pandas 顺序不一致

#### F. `tag`

问题：

- Python 使用 `np.select`，是 first-match 语义
- 条件表达式仍是 pandas `eval` 风格

当前状态：

- planner 会把它视为 rows-capable 的规划对象
- 但 Java/Flink 侧尚未真正原生实现条件表达式与 first-match 赋值

风险：

- 若后续使用自定义表达式翻译器，必须保证条件优先级和空值行为与 Python 完全一致

#### G. `formatter` / `date_formatter`

问题：

- 这两类看似更适合原生实现，但仍有不少细节必须对齐：
  - `None/NaN` 转空字符串还是转 `null`
  - 时区本地化行为
  - 时间戳精度
  - 数值格式化字符串细节

当前状态：

- 当前已经有第一版 Java native 实现
- 但仍只建议把它视为“保守可用子集”，不是已经完成全面一致性验证的终态

风险：

- 原生实现最容易出现“看起来差不多，但细节不完全一致”的问题

## 6. 当前结果一致性是如何保证的

当前 backend-flink 的“一致性”保障方式不是：

- Java/Flink 已经完整复刻 Python

而是：

- Java 先做 canonical DSL 收口
- Java runner 动态 enrich `execution` 并切 segment
- 能 native 的段先用 Java/Flink 执行
- 不能 native 的段再通过 Python reference runner 兜底

因此当前我们可以比较诚实地描述为：

- **接口侧：Java/Flink 后端已具备**
- **协议侧：共享 DSL 已具备**
- **规划侧：runtime enrich + segment planner 已具备**
- **结果侧：当前由 Java native + Python fallback 共同保证一致**
- **Flink 原生执行侧：已经起步，但绝不能对外宣称已完全替换 Python**

## 9. 后续建议的优化顺序

建议按下面顺序推进，而不要一开始就试图“所有算子全部改成纯 Flink”：

### 第一阶段：先把 native path 做成可切换

- 在 `HybridPipelineService` 中支持：
  - 全量 Python fallback
  - 单 stage native + 其余 fallback
  - 全量 fallback debug 对照

目标：

- 先让一小部分最稳的算子真的由 Java/Flink 执行

优先建议顺序：

1. `rename`
2. `filter`
3. `formatter`
4. `date_formatter`
5. `tag`

### 第二阶段：补 `_row_id` 与稳定排序/回拼

必须先解决：

- 原始行稳定身份
- sort 后顺序恢复
- 列切分后按 `_row_id` 回拼

否则：

- native 执行很容易出现“值对了但顺序不对”

### 第三阶段：原生 aggregate

重点补：

- partial agg + final agg
- rename / 输出列顺序
- grouped 输出与 Python 对齐

### 第四阶段：原生 series_transform 子集

建议只从最小子集开始：

- `shift`
- `diff`
- `rolling`

并且：

- 必须先补表达式分类器
- 不要仅靠字符串包含关系长期维持

### 第五阶段：最后再碰 callable 类算子

最后处理：

- `col_apply`
- `col_assign.lambda`
- 复杂 `series_transform`

如果业务上强依赖这些 callable：

- 更现实的路线通常不是“完全翻译成 Java”
- 而是逐步把常用表达式抽成受限 DSL 子集

## 10. 当前对外表述建议

为了避免后续沟通误差，建议当前对业务或技术同事的说法统一为：

- `backend-flink` 已完成独立服务骨架、Mongo 存储、共享 DSL 归一化、stage planner、Python 兼容执行桥。
- 当前结果一致性主要依赖 Python reference runner 兜底。
- Java/Flink 原生执行仍在逐步替换阶段，尚未完成所有算子的同义实现。
- 对于 `col_apply`、`col_assign.lambda`、复杂 `series_transform` 等强 Python callable 场景，当前仍不应宣称已具备纯 Java/Flink 完整兼容能力。

## 11. 一句话总结

本次 `backend-flink` 的成果，本质上是把“未来可替换成 Flink 原生执行”的骨架搭起来了，并先用 Python reference runner 把结果一致性守住了。

当前最重要的现实边界是：

- **Java 服务已可用**
- **Flink 规划已开始**
- **Python 语义已对齐**
- **但 Java/Flink 原生执行还没有真正完成对 Python 的全量替代**
